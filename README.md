# YarguMark

**YarguMark** — инструмент для поиска в текстах упоминаний субъектов из российских реестров (иноагенты, нежелательные и террористические / экстремистские организации, запреты судов) и для **юридически аккуратной HTML-разметки** этих фрагментов.

Всё рабочее состояние живёт **в одной локальной SQLite**: скачанные страницы, сущности реестров, алиасы, результаты языковой модели, кэш вызовов LLM, извлечённые спаны, итоговые упоминания и кэш отрисовки. Можно **докачивать новости**, **пересинхронизировать реестры**, **пересчитывать разметку** — без потери уже сделанной работы.

### Зачем это удобно

- **Один раз** прогоняете обход сайта и NLP по корпусу: это самая тяжёлая и дорогая по токенам фаза (извлечение сущностей из текста через модель).
- Дальше **обновления реестра** (в том числе когда список экстремистов / иноагентов меняется) сводятся к **`yargumark-registry-sync`** и **`yargumark-reindex`**: пересборка упоминаний идёт **из уже сохранённых спанов и актуального реестра**, **без повторного прогона LLM по всем страницам**. Это быстро и дёшево.
- Повторный NLP по **тому же тексту статьи** не бьёт по API: ответ кэшируется по хэшу тела страницы.
- Есть **веб-интерфейс** на Streamlit: библиотека новостей, документ, редактор реестра, сводки по стоимости и т.д.
- Модель по умолчанию — **Anthropic**; идентификатор задаётся в `ANTHROPIC_MODEL`. Другой провайдер LLM потребует замены слоя вызовов в коде NLP.

## Требования

- Python **3.12+**
- [**uv**](https://docs.astral.sh/uv/)
- ключ **Anthropic API** для сценариев с извлечением и обогащением алиасов

## Окружение

```bash
cp .env.example .env
```

| Переменная | Назначение |
|------------|------------|
| `ANTHROPIC_API_KEY` | доступ к API |
| `ANTHROPIC_MODEL` | модель (по умолчанию в репозитории: `claude-haiku-4-5-20251001`; можно другую из линейки Anthropic под ваш ключ) |
| `DB_PATH` | путь к SQLite (по умолчанию `data/yargumark.db`) |
| `MODE` | `demo` / `production` — пороги в UI |
| `DEMO_CONFIDENCE_THRESHOLD` / `PRODUCTION_CONFIDENCE_THRESHOLD` | пороги матчера |
| `CONTEXT_CHECK_LOW` / `CONTEXT_CHECK_HIGH` | опциональная доп. проверка для пограничных персон |

Полный список — в `.env.example`.

## Жизненный цикл

**Первичная настройка:** установить зависимости, создать схему БД, залить реестры, при необходимости обойти сайт, прогнать NLP по документам, выполнить reindex.

**Регулярное обслуживание:** подтянуть новые URL краулером (или точечно обработать новые `doc-id`), для свежих текстов — снова `process-doc`. Когда **изменились только списки реестров** — `registry-sync` и **`reindex`**; LLM по старому корпусу не нужен.

## Команды (порядок для «с нуля»)

1. `uv sync`
2. `uv run yargumark-init-db`
3. `uv run yargumark-registry-sync sync`
4. `uv run yargumark-crawl` — опции: `--start-url`, `--max-depth`, `--link-limit`, `--only-new`, `--fast-local` (см. `src/yargumark/crawler/run.py`)
5. `uv run yargumark-process-doc --doc-id N` или `uv run yargumark-process-doc --all --source uniyar --limit 500`
6. `uv run yargumark-reindex`
7. `uv run streamlit run src/yargumark/app/main.py`

Дополнительно:

```bash
uv run python scripts/seed_demo.py          # демо-данные в БД
uv run yargumark-enrich-aliases --help      # LLM-алиасы к реестру (опционально)
uv run python scripts/benchmark.py          # сверка с фикстурами
```

## Справочник CLI

| Команда | Назначение |
|---------|------------|
| `yargumark-init-db` | схема SQLite |
| `yargumark-registry-sync` | синхронизация реестров в БД |
| `yargumark-crawl` | краулер новостей |
| `yargumark-process-doc` | извлечение (LLM) + матчинг |
| `yargumark-reindex` | пересборка упоминаний без LLM |
| `yargumark-enrich-aliases` | обогащение алиасов через LLM |
| `yargumark-backfill-titles` | заголовки из сохранённого HTML |

## Стоимость NLP

Ориентиры по токенам и долларам — в **`src/yargumark/pricing.py`**. **Объём токенов** в основном от промптов и длины текста; **$/MTok** возьмите из прайса **вашей** модели ([Anthropic](https://www.anthropic.com/api) и т.д.) и подставьте в константы или в аргументы `estimate_llm_usd`.

- **Извлечение по страницам:** примерно один вызов LLM на **уникальное тело** статьи; повтор того же текста идёт из кэша.
- **Обогащение алиасов:** один вызов на сущность при первом обогащении имени; дальше кэш.
- **Reindex после смены реестра:** без новых вызовов LLM по корпусу.
